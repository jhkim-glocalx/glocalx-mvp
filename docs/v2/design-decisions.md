## Problem Statement

Korean SMB store owners need local marketing (Google Business Profile,
Instagram) but will not operate marketing software themselves. GlocalX v1
built the full self-serve loop — OAuth, Naver extraction, GBP setup, AI post
generation, publishing — and the founders concluded owners won't drive it.
v2 splits the product: a minimal owner app (Google login, GBP org-access
grant, corner chat assistant, photo+brief intake, go/no-go approval) and an
internal admin dashboard where operators + AI produce material and publish
to multiple platforms from the org account. The service delivers the
outcome; the owner touches only upload and approval.

## Demand Evidence

Thin, and named as such. The evidence is negative-shaped: v1 exists and the
founders' read is that owners won't self-serve (consistent with the CEO's
June scope alignment toward minimal-effort chat-first flows). No named
owner, no payment, no usage panic yet. The parallel-serve amendment
(below) exists precisely to convert this from belief to behavior within
two weeks: 2-3 friendly stores served manually, real photos in, real posts
published, real response latencies measured.

## Status Quo

Owners either don't market at all (stale GBP listings, photos in camera
rolls) or juggle three consoles (GBP, Naver SmartPlace, Instagram) after
close. Agencies exist but are priced and shaped for bigger businesses.
The practical current workaround for a Korean store owner is "a nephew
with Canva" or nothing.

## Target User & Narrowest Wedge

Korean local store owner (café, restaurant, salon) with a claimable GBP
listing and a camera roll of store photos. The wedge: they hand over
photos + one sentence of intent; GlocalX returns a finished post for
one-tap approval and publishes it everywhere it should go. The first
cohort is 10-20 hand-picked stores served by one operator.

## Constraints

- Only Codex writes application code (PM/verify workflow in CLAUDE.md).
- Next.js 16 canary pinned; Vercel serverless + Neon pooling (no
  LISTEN/NOTIFY, no long-lived connections) — hence polling chat.
- `APP_INTEGRATION_MODE=stub` must keep both apps fully demoable without
  live credentials.
- Reviews & performance sections stay stub-backed in v2.
- No fully automatic posting: owner approval gates every publish.

## Premises

1. **GBP org-access grant** (owner grants the GlocalX org account manager
   access) — AGREED WITH MITIGATION: operator-assisted grant via chat is
   the *primary* path for the first cohort; the self-serve flow is the
   scaling path. `pending` is a first-class, long-lived state with
   dashboard visibility and chase notes.
2. **In-app chat beats KakaoTalk for CS** — AGREED WITH KILL METRICS
   (Codex challenged this; resolution below): ship the widget as planned,
   instrumented with weekly activation, median owner response time,
   approval completion rate, and owner-initiated conversation count.
   Instrumentation sources: `cs_messages` timestamps (response time),
   `campaign_review_events` (approval completion), `activity_events`
   (activation), `cs_conversations` (owner-initiated count). Kill
   thresholds: the premise dies if Kakao's median response time is ~2x
   faster, or Kakao's approval completion exceeds the widget's by ≥20
   percentage points; activation and owner-initiated count are
   context-only signals with no kill threshold. Comparison method: the
   manually-served stores migrate to the widget when the cohort onboards,
   giving each a before/after window; the week-4 evaluation compares each
   migrated store's Kakao weeks against its widget weeks (same store,
   both arms) rather than comparing different stores. During the Kakao
   weeks, operators log every exchange in a shared sheet — owner message
   timestamp, reply timestamp, approval asked/answered/outcome — using
   the same metric definitions as the widget instrumentation, so the
   baseline is measured, not remembered. If the premise dies, v2.1 adds
   Kakao notify/CS.
3. **Build-then-serve sequencing** — AMENDED TO PARALLEL-SERVE: 2-3
   friendly stores are served manually from week 1 (v1 app + operator
   hands) while the build proceeds; their real requests become the
   dashboard's test data and the premise-1/premise-2 evidence.
4. **Owners won't self-serve** (the reason v2 exists) — accepted; v1's
   own trajectory is the evidence.
5. **v1 reuse is real** — verified in code: publishing pipeline (GBP +
   Instagram, idempotent, history-preserving), GBP state machine, OAuth,
   adapter boundary all carry over.

## Approaches Considered

### Approach A: Thin Slice (one app, /admin routes)
No monorepo split; admin route group + separate admin auth inside the
existing app. Effort M (human ~3 wks / CC+Codex ~1.5 wks), Risk Med.
Fastest to a served customer, zero restructure risk — but org publishing
credentials and operator tooling ship in the public app's deployment,
session isolation is convention rather than boundary, and the split gets
re-done later under cohort load.

### Approach B: The v2 plan, amended (RECOMMENDED)
Monorepo, two Next.js apps, two Vercel projects, shared Neon Postgres, six
phases (docs/v2/delivery-plan.md) — plus this session's four amendments:
(1) parallel-serve 2-3 stores from week 1; (2) operator-assisted GBP grant
as primary path; (3) chat kill metrics with the manual cohort as
comparison arm; (4) "AI drafts, human sends" as the chat starting posture,
making autonomous AI mode a flip rather than a build. Effort L (human ~6
wks / CC+Codex ~3-4 wks), Risk Low-Med.

### Approach C: Cockpit-first reorder
Same end state as B but dashboard built first against the unchanged v1
app; owner-app changes last. Tools operators by week 2, but forces early
CS onto KakaoTalk — quietly pre-deciding the premise-2 experiment — and
delays the intake/approval loop the service promise depends on.

## Recommended Approach

**Approach B.** The separation decision was already made deliberately for
credential isolation; the plan documents (docs/v2/README.md,
architecture.md, delivery-plan.md) exist with acceptance criteria; and the
four amendments close the plan's two real weaknesses — no customer contact
until week 6 (fixed by parallel-serve) and an unmeasured chat bet (fixed
by kill metrics). Phase 0 (the split) produces no customer value and must
stay strictly time-boxed at one week.

Four session amendments plus one reconciliation to fold into docs/v2/
before implementation:
- delivery-plan.md Phases 1 and 3: add the chat kill metrics to
  acceptance criteria (definitions and thresholds in Premises §2).
  Response time, activation, and owner-initiated count instrument in
  Phase 1 (`cs_messages`, `activity_events`, `cs_conversations`);
  approval completion rate activates in Phase 3 when
  `campaign_review_events` lands — Phase 1 cannot measure it.
- delivery-plan.md Phase 2 + architecture.md §2/§5: starting posture is
  "AI drafts, operator sends." This is a third conversation state the
  current two-mode schema cannot express — it requires a `draft` status
  on `cs_messages` (AI-composed, not owner-visible), a draft
  review/edit/send surface in the console, and a send action. This adds
  scope to Phase 2 (roughly +2-3 days); autonomous `ai` mode then becomes
  a per-conversation flip once draft quality is trusted.
- delivery-plan.md Phase 4: operator-assisted grant is the primary
  onboarding path; the self-serve flow is the scaling path.
- delivery-plan.md: add a Phase 0-parallel workstream — "serve 2-3 stores
  manually from week 1" with an explicit owner (founder), not engineering.
- (housekeeping, from review — not a session amendment) README.md §6,
  architecture.md §1, delivery-plan.md (working model + Phase 0
  acceptance): reconcile all `dev`-branch references with GitHub Flow — `feat/social-post-publishing` carries commit f89fbc0 which
  retires the `dev` branch, so Phase 0's "dev-branch Vercel preview"
  acceptance criterion and the branch-mapping tables must be rewritten
  against `main` + per-PR previews before Phase 0 wires the second
  Vercel project.

Phase 0 overrun rule: if the split exceeds one week, cut the packages
extraction to `packages/db` only (admin app imports domain/integrations
from the owner app via path alias temporarily) and defer the rest to a
cleanup PR — do not let restructure work eat Phase 1.

## Success Criteria

- A new store reaches `granted` GBP access with zero engineer involvement
  (operator involvement allowed and expected).
- Owner upload → multi-platform publish in under one business day, owner
  touching only upload and go/no-go.
- One operator serves the 10-20 store cohort from the dashboard alone.
- Chat kill metrics collected weekly from week 1 of cohort service;
  premise 2 explicitly re-evaluated at week 4.
- Both apps fully demoable in stub mode.

## Distribution Plan

Web services on the existing Vercel pipeline (two projects). Branch
mapping follows GitHub Flow once `feat/social-post-publishing` lands:
`main` → production, per-PR previews for review (no `dev` branch). No
new artifact types — existing deployment pipeline covers this.
